Performance · §

One backend, two ways to drive it

A first look at Chimera's pNFS data path. Two diskfs storage servers in a flex-files pNFS layout, one client, and the exact same fio workload run two ways: once through the stock Linux kernel NFS mount, and once through Chimera's in-process fio engine that speaks pNFS over RDMA directly from the application. Same servers, same fabric — the client is the only variable.

Note: single-client results from a development rig (June 2026), not an audited benchmark. Every number below is taken straight from the fio output, which is reproduced in full — along with the client and server configs — so the runs can be repeated. Treat them as directional.

// how to read this

Both runs drive the same two servers, the same flex-files pNFS layout, and the same 400 GbE fabric with the same fio workload. The only thing that changes is the client. They are not, however, an apples-to-apples product comparison — they deliberately measure two different things:

  • Linux NFS is the stock in-kernel client: general-purpose, POSIX, page-cache backed, available to any unmodified process. Its small-IO ceiling here — ~1M IOPS, ~4 GB/s, about 8% of line rate — isn't the wire or the servers; it's the kernel client's software model. Every fio submission is handed off to the shared nfsiod / rpciod worker pool, so the run shows 17–22 million context switches and dozens of kworkers at 20–32% CPU. That hand-off, plus the shared RPC state behind kernel spinlocks, means a large fraction of the box goes to context-switching and cross-core lock contention rather than to moving data.
  • Chimera's fio engine is a purpose-built in-process client that speaks pNFS/RDMA directly from the application's own threads — no syscall, no worker hand-off, no kernel context switch on the I/O path. Critically, the client keeps its RPC and session state thread-local: each fio thread drives its own connection with no shared structures and no spinlocks, so adding threads adds throughput instead of lock contention. The result is ~840 context switches over the whole 30 s run and the client link driven to ~86–89% of 400 GbE line rate. The cost is CPU spent busy-polling: fio runs at 3128% (~31 cores) — but those cores are doing I/O, not spinning on locks.
  • So read this as "what the wire and the servers can deliver once the client software gets out of the way" versus "what today's kernel NFS client delivers to an unmodified application." Both are legitimate; they answer different questions.
  • The layout group is a short warm-up (file creation, 0.6–1.1 s) absorbed by the servers' write cache — the steady-state comparison is the timed 30 s random phases.
  • The random-read working set (~25.6 GiB) fits in the servers' block cache, so the read figures reflect cache + RDMA fabric, not cold NVMe media; the 30 s random-write window is likewise absorbed by write-back cache.

Client-side top during each run — this is the clearest view of the difference. Under the Linux mount the CPU is spread across a swarm of nfsiod / rpciod kworkers (the kernel's shared RPC machinery); under Chimera it is a single fio process at ~3128% with no kernel I/O threads at all, because the client's state is thread-local and never touches that pool. Headers and busy rows are verbatim; the long tail of idle kernel threads is elided, as marked.
