Status & conformance · §

Project Maturity

The NFSv3, NFSv4 (4.0 / 4.1 / 4.2), and SMB2/3 servers implement the full data-path and metadata operation set and pass real-Linux-client conformance suites. The data path — read, write, metadata, locking — is mature and well exercised. NFSv4 reboot-recovery state and SMB durable handles now persist across restarts, and NFSv3 gained NSM/statd lock recovery. Kerberos for NFS and several advanced protocol features remain present-but-incomplete or absent. Those gaps are listed below.

§ 01 At a glance

Four protocol servers over one pluggable async VFS. Because they share the VFS, all of them work across every backend — in-memory, Linux passthrough, and persistent.

NFSv3 Mature

All 21 procedures, full MOUNT v3, and a real NLM byte-range lock manager routed through the unified VFS lease layer. TCP and RPC-over-RDMA.

NFSv4 · 4.1 · 4.2 Mature core

Single COMPOUND handler across all three minor versions. Sessions, state, ACLs, xattrs, delegations, and pNFS (flex-files). Reboot-recovery records and the reply cache now persist to the VFS — clients and state survive a restart.

SMB2 · SMB3 Broad coverage

All 19 commands, dialects through 3.1.1, signing, encryption, oplocks & leases (file and directory), durable / persistent handles, change-notify, transport compression, copy-offload, ACLs, Kerberos. Several advanced features ship disabled by default.

S3 Objects + buckets

Object data path — GET/PUT/HEAD/DELETE, ranged reads, full multipart, SigV4/V2 — plus dynamic bucket create/delete/list and paginated prefix/delimiter listing. Versioning, ACLs, SSE, and conditional requests are absent.

§ 02 Backends

Every protocol sits on the same VFS, so it runs identically across all backends. A handful of features are backend-specific and noted in the per-protocol detail.

§ 03 Feature detail, protocol by protocol

Each protocol below splits into what's done, what's partial (works with caveats), and what's missing. This is a summary — for the line-by-line detail and known bugs, the GitHub issue tracker is the source of truth.

NFSv3 + MOUNT v3 + NLM v4 locking
Mature

✓Done

  • All 21 procedures implemented, no stubs
  • MOUNT v3 — MNT, DUMP, UMNT, EXPORT, in-memory mount table
  • NLM v4 advisory byte-range locking through the unified VFS lease layer, with cross-protocol conflict detection
  • NSM / statd (prog 100024) — crash-recovery lock reclaim, with a KV-persisted monitor list and state number
  • NLM lock state persisted per-host across a server restart
  • Write stability & COMMIT with per-instance write verifier
  • Weak cache consistency (wcc) pre/post attrs
  • Duplicate Request Cache (v3 path, config-gated)
  • TCP and RPC-over-RDMA transports; built-in portmap / rpcbind

~Partial

  • Blocking locks: cross-protocol conflicts ride the wait path; same-protocol contention returns DENIED (no async GRANTED callback queue)
  • SHARE / UNSHARE (DOS share modes) accepted but not enforced
  • EXPORT access control: advertised to all clients, no per-client restriction

−Missing

  • RPCSEC_GSS / Kerberos — non-AUTH_SYS flavors fall back to anonymous
  • Per-export root_squash / per-client access control
  • NFSv3 ACL sideband protocol (prog 100227)
  • UDP transport (TCP / RDMA only)
NFSv4 · 4.1 · 4.2 single COMPOUND handler, minorversion 0–2
Mature core

✓Done

  • 4.0 base — every RFC 7530 operation has a real handler on the async VFS
  • 4.1 sessions & state — EXCHANGE_ID, CREATE_SESSION, SEQUENCE, RECLAIM_COMPLETE, layout ops
  • 4.2 — ALLOCATE, DEALLOCATE, SEEK, intra-server COPY, RFC 8276 xattrs
  • Reboot recovery — recovery records and the reply cache persist to the VFS KV store; clients and lock/open state are remembered across a restart, with a real grace window
  • Delegations — read/write, callback channel, recall timeout (config-gated, default off)
  • OPEN claims incl. CLAIM_DELEG_CUR_FH; BIND_CONN_TO_SESSION
  • pNFS — MDS + DS, flex-files primary; block & SCSI layout types
  • Full NFSv4 ACL model end-to-end
  • v4.0 duplicate-request cache keyed by XID

~Partial

  • SET_SSV accepts but returns an empty digest (SSV-secured RPC not wired)
  • SECINFO_NO_NAME returns a single hardcoded flavor
  • pNFS block / SCSI layout types are scaffolded; flex-files is the production path

−Missing

  • RPCSEC_GSS / Kerberos — SECINFO advertises a krb5 OID but the RPC layer only does AUTH_SYS/AUTH_NONE
  • 4.2: READ_PLUS, CLONE, WRITE_SAME, IO_ADVISE, inter-server COPY
  • GET_DIR_DELEGATION
  • OPENATTR / named-attribute directories (user xattrs served via 4.2 ops instead)
  • CLAIM_DELEGATE_PREV (delegation reclaim after client reboot)
SMB2 · SMB3 dialects 2.1 – 3.1.1 · all 19 commands
Broad coverage

✓Done

  • All 19 SMB2 commands with real handlers
  • Dialects 2.1 / 3.0 / 3.0.2 / 3.1.1, with 3.1.1 negotiate contexts (preauth integrity, signing, encryption)
  • Auth: NTLM/NTLMSSP, Kerberos (GSSAPI/krb5 + SPNEGO), guest, winbind/AD
  • Signing: HMAC-SHA256, AES-CMAC, AES-GMAC
  • Encryption: AES-128/256 CCM & GCM
  • Oplocks & leases (R/RW/RWH), including SMB3 directory leases, on VFS caching leases
  • Per-share encryption enforcement — unencrypted access to an encrypt_data share is refused with ACCESS_DENIED
  • SMB3 transport compression — LZ77, LZ77+Huffman, LZNT1, Pattern_V1, applied on the wire
  • Copy-offload (ODX OFFLOAD_READ/WRITE, DUPLICATE_EXTENTS), FILE_LEVEL_TRIM, GET/SET_INTEGRITY
  • Multi-element byte-range locks; symlink reparse (STATUS_STOPPED_ON_SYMLINK)
  • Multichannel and SMB Direct / RDMA transport; real credit windowing
  • Durable & persistent handles persisted to the backend atomic KV store; change-notify (VFS-notify backed)
  • Security descriptors / ACLs with NT↔VFS mapping; rich FSCTL set

~Partial

  • SMB Direct / RDMA: SMB 3.0 framing only — the 3.1.1 transform (encrypted) variant is not yet wired, so RDMA is unencrypted-only
  • Named streams / ADS: full on memfs; passthrough/persistent backends don't yet expose xattr-backed streams
  • SACL emitted empty (DACL/owner/group are real)

−Missing

  • DFS, Witness protocol
  • Extended Attributes (accept-and-ignore)
S3 HTTP-to-VFS bridge · object data path
Object path only

✓Done

  • Object ops: Put, Get (ranged), Head, Delete, batch DeleteObjects, server-side CopyObject
  • Multipart: full set — create, upload, complete, abort, list, UploadPartCopy — on every backend
  • Bucket ops: dynamic CreateBucket / DeleteBucket / HeadBucket / ListBuckets (backed by a real VFS directory)
  • Listing: prefix, delimiter / CommonPrefixes, V1 marker and V2 continuation-token pagination
  • Auth: AWS Signature V4 and V2; streaming (aws-chunked) uploads
  • Keep-alive; structured XML errors

~Partial

  • ETag is not content-MD5 (it's a hash of size/mtime/handle) — clients validating ETag==MD5 will mismatch
  • Streaming uploads: chunk framing is decoded but per-chunk chunk-signatures are not verified
  • DeleteBucket is non-recursive — a non-empty bucket returns BucketNotEmpty
  • Single root credential — key is authenticated but not mapped to a VFS identity (no per-user authz)
  • No presigned URLs / anonymous access

−Missing

  • Versioning — ListObjectVersions is a null-version stub; no real version history
  • Bucket policy / ACL / lifecycle / CORS / tagging
  • Object ACL, tagging, object lock / retention
  • Server-side encryption (SSE / SSE-KMS / SSE-C)
  • User metadata (x-amz-meta-*); conditional requests
REST management API HTTP/HTTPS admin · OpenAPI 3.0 + Swagger UI
Early v1

✓Done

  • Runtime CRUD for NFS exports, SMB shares, S3 buckets, VFS mounts, and local users
  • GET /version, read-only GET /api/v1/config
  • Auth: JWT Bearer (HS256) and HTTP Basic (crypt + optional winbind/AD)
  • HTTP and HTTPS/TLS via libevpl; bundled Swagger UI + OpenAPI spec

~Partial

  • Mutations are in-memory — not written back to config, lost on restart
  • No update/patch verbs (create + delete only)
  • JWT signing secret is generated per process; tokens don't survive a restart

−Missing

  • RBAC — any authenticated user is a full admin
  • Config write-back / persistence of changes
§ 04 Authentication, in one place
security: read this if you’re evaluating

Strong authentication (Kerberos / GSSAPI) is implemented and working for SMB, but not for NFS — the NFS RPC layer processes AUTH_SYS / AUTH_NONE only, and the krb5 advertisement in SECINFO is not yet implemented.

On the S3 side the access key is authenticated but not mapped to a VFS identity, so there is no per-user authorization yet — all operations run as root.

// the short version

The data-path and metadata operations across NFSv3, NFSv4, and SMB2/3 are mature and pass real-client conformance suites. NFSv4 reboot-recovery state and SMB durable handles now persist across restarts, and S3 gained dynamic bucket management and full listing. The most significant gaps to know about before leaning on Chimera in production: no Kerberos for NFS; the S3 server has no versioning, ACLs, SSE, or conditional requests, and its access key isn't mapped to a per-user identity; and several advanced SMB / NFSv4 features ship disabled by default. The REST management API is an early v1 — functional but in-memory and without RBAC.

This page is a summary. For line-by-line operation support, current bugs, and the roadmap, see the issue tracker on GitHub or ask in Discord.
